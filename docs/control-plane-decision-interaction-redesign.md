# Control Plane Decision Interaction Redesign Proposal

## 1. Current interaction model analysis

### What the user sees today

The Control Plane already starts with a repository-centered card: the owner refreshes repository intelligence, sees a single **Do Next** recommendation, sees repository state/strategy/evidence/implementation guidance, can save outcome evidence, and can refresh again. However, the primary CTA is still driven by the current workflow step. Depending on the recommendation type, the owner may see labels such as **Copy Implementation Prompt**, **Open Codex**, **Mark External Work Complete**, **Validate Result**, **Prepare AI Context**, **Copy Understanding Check**, **Open ChatGPT**, **Paste Validation Response**, **Run Validation**, **Review Canonical Edit**, or **Apply Canonical Edit**.

The UI also exposes a detailed workflow progress renderer in the work item page: repository state, current step, next repository state, completed steps, remaining steps, progress percentage, and a primary workflow button. Development diagnostics disclose the workflow key, workflow index, repository state, next repository state, step id, step label, step classification, localStorage value, refresh flags, and other orchestration details.

### What the workflow engine is doing

The workflow engine deterministically selects one workflow type from recommendation metadata, expands it into a fixed checklist, persists the current step in localStorage, advances one step at a time, and refreshes repository intelligence when terminal refresh states are reached. Step classifications currently distinguish actions that require user clicks, external work, automatic bridge steps, and refresh-only steps. The application already auto-advances `auto-advance` steps and auto-starts refresh-only steps while preserving workflow diagnostics.

### Where those differ

The repository owner is trying to make and execute repository decisions. The engine is coordinating deterministic internal orchestration. Those are different models:

- The user model is **Repository → Decision → Execute → Record Outcome → Refresh → Next Decision**.
- The engine model is **Recommendation Ready → Workflow In Progress → Waiting for External Work → Validate Result → Refresh Repository**.

The current UI sometimes translates engine labels into friendlier words, but the primary visible flow is still step advancement. For example, **Open ChatGPT** does not open ChatGPT, **Paste Validation Response** does not collect pasted content, **Run Validation** is only a bridge into refresh, and the primary **Mark External Work Complete** action does not persist outcome evidence; the separate **Save Outcome** panel does.

### Where the UI exposes internal FSM implementation

The normal path exposes internal FSM details when it shows:

- Repository state and next repository state as first-class dashboard metadata.
- Workflow progress percentage and completed/remaining FSM steps.
- Manual CTAs for bridge-only steps that do not collect input or perform local work.
- Separate primary workflow advancement and outcome persistence controls for the same external-work completion concept.
- Development diagnostics adjacent to the normal Control Plane rather than nested under a clearly advanced/debug section.
- Prompt/package generation as a workflow step rather than as an automatic result of choosing an execution agent.

## 2. Redesigned interaction model

### Repository lifecycle

1. **Repository disconnected** — owner enters a local path.
2. **Repository refresh running** — Agent IDE runs deterministic local intelligence generation and streams progress.
3. **Repository intelligence ready** — generated artifacts, diagnostics, rankings, prompts, and context packages are available.
4. **Next repository decision ready** — the Control Plane presents the current repository state, highest-leverage decision, evidence, risk, and expected outcome.
5. **Execution package ready** — after the owner chooses an execution agent, Agent IDE generates/copies/opens the correct package.
6. **External work pending** — owner performs the work outside Agent IDE.
7. **Outcome recorded** — owner records success/failure/blocked/skipped evidence and optional notes.
8. **Repository intelligence refreshed** — Agent IDE regenerates repository intelligence and presents the next decision.

### User decision lifecycle

A visible decision exists only when the repository owner must choose, approve, provide, or do something that Agent IDE cannot deterministically infer:

1. **Review decision** — read the selected repository decision and why it is highest leverage.
2. **Choose execution agent** — Claude, Codex, ChatGPT, Gemini, Cursor, or Copy Package.
3. **Copy/open generated package** — package is already generated by the local deterministic UI layer once the agent is selected.
4. **Perform external work** — no FSM steps are shown while external work is happening.
5. **Record outcome evidence** — persist outcome, prompt quality, notes, and optional refresh.
6. **Approve/edit canonical intent** — only for product-decision/canonical-intelligence changes.

### Normal interaction flow

```text
Repository Refresh
  ↓
Next Repository Decision Ready
  ↓
Choose Execution Agent
  - Claude
  - Codex
  - ChatGPT
  - Gemini
  - Cursor
  - Copy Package
  ↓
Agent-specific Execution Package Generated
  ↓
Copy/Open Package
  ↓
Perform Work Outside Agent IDE
  ↓
Record Outcome
  ↓
Repository Intelligence Refresh
  ↓
Next Repository Decision Ready
```

### Advanced/debug flow

Advanced remains available but does not gate the happy path:

- Workflow diagnostics, persisted workflow state, state classifications, and localStorage debugging.
- Context Package.
- Architect Prompt.
- Builder Prompt.
- Reviewer Prompt.
- Debugger Prompt.
- Validation Prompt.
- Recommended package raw preview.
- Decision Ranking details.
- Repository Judgment/Product Judgment shadow details.
- Judgment Comparison.
- Repository Intelligence Explanation.
- Evidence Lineage and Evidence Explorer.
- Verification, quality, AI handoff validation, trend, timeline, raw intelligence, generated artifacts.

## 3. Existing FSM classification

### Repository workflow states

| Existing state | Classification | Why |
|---|---|---|
| Repository Not Connected | User Decision | The owner must provide or choose a local repository path. |
| Refresh Repository Intelligence | User Decision | Starting an initial/manual refresh is owner-triggered unless reached after outcome recording. |
| Repository Analysis Running | Automatic | The local deterministic refresh pipeline is running and should only stream progress. |
| Recommendation Ready | User Decision | The next repository decision is available for review and agent choice. |
| Workflow In Progress | Automatic | This is an orchestration bucket, not a repository-owner decision. |
| Waiting for External Work (Codex / ChatGPT / User) | External Work | The owner or external agent is doing work outside Agent IDE. |
| Validate Result | Automatic | Today it is a deterministic bridge to refresh unless a future visible validation input is introduced. |
| Refresh Repository | Automatic | Once a workflow reaches refresh, the refresh should start without another FSM click. |
| Complete | Advanced / Diagnostics | Completion is useful for diagnostics/history, but the normal user model should show the next decision or up-to-date state. |
| Next Recommendation Ready | User Decision | This is the next visible repository decision after refresh. |

### Workflow step ids

| Step id | Classification | Why |
|---|---|---|
| copy-context-package | Advanced / Diagnostics | Context Package must remain available, but normal package generation should happen after agent selection rather than as a required FSM step. |
| copy-validation-prompt | Advanced / Diagnostics | The validation prompt remains available under Advanced; normal flow should generate the selected package automatically. |
| copy-understanding-check | Advanced / Diagnostics | It is an artifact-copy step, not a repository decision. |
| copy-implementation-prompt | Automatic | The implementation package should be generated automatically after execution-agent selection; user interaction is copy/open, not prepare. |
| review-canonical-edit | User Decision | The owner must review generated canonical intent before it becomes repository intent. |
| edit-proposal | User Decision | Editing/approving owner-authored decision text is real owner input. |
| apply-canonical-edit | User Decision | Writing canonical intent requires owner approval. |
| review-question | User Decision | A blocking question requires owner judgment. |
| inspect-evidence | Advanced / Diagnostics | Evidence remains available, but inspecting it should not be a mandatory FSM advancement step. |
| record-finding | User Decision | The owner must record new repository knowledge/findings. |
| review-documentation-gap | User Decision | The owner reviews the proposed documentation decision. |
| edit-documentation | External Work | Documentation edits happen outside the deterministic preparation layer. |
| review-diff | User Decision | Reviewing changed documentation is an owner decision before outcome/refresh. |
| open-codex | Automatic | The current behavior copies a prompt and advances state; opening/copying belongs to the generated execution package action. |
| run-implementation | External Work | The actual implementation is outside Agent IDE; completion should be recorded through outcome evidence. |
| open-chatgpt | Automatic | Current implementation performs no local work and collects no input; it should be represented as part of package launch/copy. |
| paste-response | External Work | If response evidence is actually required, it belongs in outcome/evidence recording; otherwise it should disappear from normal flow. |
| validate-result | Automatic | Bridge-only validation should auto-progress to refresh while diagnostics remain available. |
| run-validation | Automatic | Current behavior is refresh orchestration, not a separate validation action. |
| refresh-repository | Automatic | After outcome/evidence is recorded, refresh should start automatically when selected. |

## 4. Existing UI component mapping

| Component/area | Proposed treatment | Why |
|---|---|---|
| Repository connection card | Remain unchanged with copy tweaks | Local path entry and manual refresh are legitimate owner actions. |
| Refresh progress list | Remain unchanged | Streaming deterministic local progress is useful feedback. |
| CurrentTaskCard / Do Next | Remain and become primary | This is already the repository-decision surface and should replace workflow progress as the main path. |
| RepositoryDecisionAnswers | Remain and become primary | The four-question structure explains repository state, rationale, decision, and execution. |
| WorkflowPrimaryButton | Replace in normal workflow | The single FSM-step CTA should become decision actions: choose agent, copy/open package, record outcome, refresh. |
| WorkflowProgress | Move into Advanced | Progress through internal states is diagnostic, not normal owner workflow. |
| WorkflowDiagnosticsDisclosure | Move into Advanced / Diagnostics | Preserve all diagnostics but keep them out of the happy path. |
| CompletionPanel | Remain and become the canonical completion control | Outcome persistence is the real user action; it should replace state-only “Mark External Work Complete.” |
| WorkItemPage | Redesign as Execution Package page | It should show selected decision, agent picker, generated package, acceptance criteria, and outcome recording; workflow internals move to Advanced. |
| ValidationWorkspace | Move into Advanced plus package generator | Validation prompt/context remain available, but the normal flow should present validation as an execution package. |
| CanonicalEditPanel | Remain in normal flow for product decisions | Owner approval/editing of canonical intent is explicitly user-facing. |
| Recommended Package details | Move under Advanced and package preview | Raw package preview remains useful, but normal flow should be agent-selection driven. |
| Quick Actions | Move into Advanced or secondary utilities | Useful shortcuts, not part of repository decision execution. |
| Advanced Repository Intelligence disclosure | Remain and expand | This becomes the home for generated artifacts, diagnostics, ranking details, prompts, and raw intelligence. |
| ShadowRecommendationCard / ProductJudgmentShadowCard / RepositoryJudgmentReadinessCard | Move/remain in Advanced | Shadow engines are diagnostics/intelligence, not normal owner decisions. |
| Decision Ranking details | Move/remain in Advanced | Ranking is important for auditability, but only the selected decision should drive the happy path. |
| Repository Health / Understanding / Risks | Remain summarized, details in Advanced | High-level state helps context; detailed grids should not crowd execution. |
| Verification / Quality / AI Handoff Validation | Move into Advanced except warnings | Diagnostics are preserved; only failures/warnings should surface in the normal path. |
| Prompt Center sidebar section | Remain as Advanced prompts | Requirement explicitly preserves all prompt artifacts. |
| Context Package sidebar section | Remain as Advanced handoff | Requirement explicitly preserves the Context Package. |
| Architecture / Strategy / Backlog / Validation / Decisions sections | Remain unchanged | Generated intelligence artifacts remain browseable. |

## 5. Implementation roadmap

### PR 1 — Add a decision-flow adapter and proposal documentation

- **Motivation:** Introduce the user-facing model without changing workflow semantics or intelligence outputs.
- **Files changed:** New decision-flow types/selectors near the UI layer; tests for mapping current recommendations/workflows to decision stages; this proposal document.
- **Testing:** `npm test`, `npm run build`, focused tests asserting recommendation and workflow outputs are unchanged.
- **Rollback risk:** Low. The adapter is read-only and can be removed without touching the engine.

### PR 2 — Replace the primary Control Plane CTA with decision actions

- **Motivation:** Make the dashboard say what the owner needs to do: choose agent, copy/open package, record outcome, refresh.
- **Files changed:** `src/App.tsx`, style updates, tests for visible labels and copy behavior.
- **Testing:** `npm test`, `npm run build`, UI text tests confirming FSM-only labels are not on the happy path.
- **Rollback risk:** Medium-low. It changes presentation, but can fall back to the old workflow CTA because engine state remains intact.

### PR 3 — Add execution-agent package selection

- **Motivation:** Automatically generate an agent-specific package after agent choice and remove preparation-state clicks.
- **Files changed:** UI package-generation component, deterministic package label/content selector, tests for Claude/Codex/ChatGPT/Gemini/Cursor/Copy Package options.
- **Testing:** `npm test`, `npm run build`, snapshot/fixture tests showing base prompt content is unchanged except deterministic wrapper labels.
- **Rollback risk:** Medium. Clipboard/open flows are user-visible, but all generated prompt artifacts are preserved.

### PR 4 — Collapse bridge-only steps from the normal path

- **Motivation:** Auto-progress deterministic bridge states and hide `Workflow In Progress`, `Validate Result`, and refresh-only states from normal interactions.
- **Files changed:** UI orchestration layer around `advanceWorkflow`, normal-path rendering, diagnostics placement, tests for auto-advance behavior.
- **Testing:** `npm test`, `npm run build`, workflow-engine tests proving internal semantics are unchanged.
- **Rollback risk:** Medium. It affects timing of UI state changes, but existing diagnostics and manual fallback remain in Advanced.

### PR 5 — Consolidate external-work completion around outcome recording

- **Motivation:** Eliminate confusion between state-only “Mark External Work Complete” and persistent “Save Outcome.”
- **Files changed:** `CompletionPanel`, Control Plane dashboard/work item page, outcome-related tests.
- **Testing:** `npm test`, `npm run build`, outcome save tests verifying `.ai/outcomes.json` and `.ai/outcomes.md` behavior remains deterministic.
- **Rollback risk:** Medium. Outcome recording is important, but the API and file semantics remain unchanged.

### PR 6 — Reorganize Advanced/debug sections

- **Motivation:** Preserve power-user visibility while keeping normal flow simple.
- **Files changed:** Advanced disclosure composition, Prompt Center/Context Package links, diagnostics relocation, style updates.
- **Testing:** `npm test`, `npm run build`, UI tests ensuring required artifacts remain accessible.
- **Rollback risk:** Low-medium. Mostly presentation reorganization.

### PR 7 — Final compatibility hardening

- **Motivation:** Lock in deterministic behavior and backward compatibility after the UI redesign.
- **Files changed:** Regression tests, docs, possibly migration guards for old localStorage workflow state.
- **Testing:** `npm test`, `npm run build`, fixture comparisons proving repository intelligence outputs, recommendations, workflow semantics, generated prompts, and diagnostics remain stable.
- **Rollback risk:** Low. Test/documentation-focused.

## 6. Recommended first pull request

The highest-leverage first PR is **PR 1: Add a decision-flow adapter and proposal documentation**.

It should come first because it creates a stable seam between the deterministic workflow engine and the user-facing decision model. The workflow engine can keep producing the same states and persisted data, while the adapter exposes a repository-owner vocabulary: repository status, selected decision, required owner action, execution-agent options, generated package availability, external-work status, outcome-recording status, and refresh readiness. That seam lets later PRs change presentation and orchestration incrementally without modifying recommendation generation, ranking, repository judgment, product judgment, canonical semantics, or workflow-engine semantics.

This also gives the team a testable compatibility contract before changing visible behavior: identical repository inputs should continue producing identical intelligence artifacts, recommendation selections, rankings, generated prompts, and workflow transitions; only the normal Control Plane presentation changes.
